## 발견사항

### [HIGH] 1. `getPassivePool()` 이름·구현·SoT 불일치 — 시드 vs 런타임 분리

- 위치: `lib/db/queries/items.ts:36-42`, `lib/db/seed.ts:221-234` (`seedItemPassivePool`), `lib/db/schema.ts:94-99` (`item_passive_pool`), `app/api/inventory/route.ts:91`
- 현상:
  - 함수 이름은 `getPassivePool`이고 시드 헬퍼는 `seedItemPassivePool`인데, 실제 SQL은 `SELECT id, name, description FROM skill_table WHERE is_active = 1`. 즉 모든 active skill을 패시브 풀로 사용한다 (active 스킬도 포함).
  - 가챠 라우트(`route.ts:91`)가 이 결과를 패시브 풀로 받아 `[스킬명]` 형태로 옵션에 새기므로, 결과적으로 액티브 스킬도 장비 옵션에 들어갈 수 있다. battle.ts의 `applyItemPassives`는 이름 매칭으로 invested+1을 하므로 액티브 스킬도 invested 가산되어 트리거 가능 (예: `[연속공격]`이 장비에 박힐 수 있음).
- 시나리오:
  1. 가챠로 SSR/UR 등급에서 패시브 슬롯이 1개 추가됨
  2. `passives` 풀이 `skill_table` 전체 → `passives[Math.floor(Math.random() * 18)]`에서 액티브 스킬 (예: `ACTIVE_RAGE_01` "분노")가 뽑힘
  3. 옵션 `[분노]`로 저장 → `applyItemPassives`로 인해 invested 가산 → 전투에서 "전투 시작" 트리거가 무료로 발동
- 원인: 마이그레이션 `item_passive_to_skill_v1` (schema.ts:622-627)에서 옛 패시브 이름들을 새 스킬 이름으로 교체하는 정책 결정과 함께, 패시브 풀의 SoT를 `skill_table`로 옮긴 것. 단, 함수 이름과 시드 함수, schema의 `item_passive_pool` 테이블이 모두 그대로 남아있어 코드 의도가 모호해졌다.
- 영향:
  - 의도가 "active 스킬도 가챠로 얻을 수 있다"라면 OK이지만, 그 경우 `seedItemPassivePool` + `item_passive_pool` 테이블은 dead. 데이터베이스에 옛 이름 4개가 영구 잔존.
  - 의도가 "패시브 스킬만"이라면 SQL에 `WHERE type='passive'` 필터 누락 — 액티브 스킬이 옵션에 박히는 silent 버그.
- 제안:
  - 의도 결정 후 둘 중 하나:
    - (A) 패시브만: `getPassivePool()`에 `AND type='passive'` 추가
    - (B) 모든 스킬 허용: `getPassivePool()` → `getEquippableSkillPool()` 등으로 rename + `seedItemPassivePool` 제거 + schema에서 `item_passive_pool` 제거 (또는 deprecated 주석)
  - 어느 쪽이든 `item_passive_pool`의 옛 데이터는 잔여물로 남으므로 마이그레이션 추가 권장 (DELETE 또는 동기화)

### [HIGH] 2. `handleEquipFromStash` 두 단계 PATCH 사이 current_hp 영구 손실 가능

- 위치: `components/game/ItemsTab.tsx:118-124`, 백엔드 `app/api/inventory/route.ts:65-71`
- 현상: 보관함에서 장비 교체 시 클라이언트가 PATCH를 두 번 호출:
  1. `delete` cur (현재 장착 중인 아이템) — `deleteEquipment`는 장착 여부와 무관하게 즉시 DELETE
  2. `equip` newItem
  각 PATCH 끝에서 백엔드는 `if (char.current_hp > effMaxHp) updateCharacter({ current_hp: Math.min(...) })`를 수행 (`route.ts:66-70`).
- 시나리오:
  1. HP 100/100 상태, 무기 장착 중 (이 무기에 HP증가 +50 옵션이 있어 max=100)
  2. 보관함의 다른 무기로 교체 → 1차 PATCH (`delete` cur): 이 시점 effective max는 무기 효과 빠진 상태(예: 60). 백엔드가 current_hp(100)을 60으로 캡 — 영구 손실
  3. 2차 PATCH (`equip` newItem): 새 무기에 HP증가 +40이 있어 max가 90이지만 current_hp는 60 그대로
  - 의도와 다르게 약 30HP 손실
- 원인:
  - `handleEquipFromStash`가 두 PATCH를 분리하여 호출하는데, 백엔드는 두 호출 사이의 임시 상태에서도 current_hp 캡을 수행
  - 한편 `handleReplace` (가챠 결과 교체) 흐름은 순서가 반대(equip new → delete old)라 같은 문제가 안 생김
- 제안:
  - (A) 백엔드: `delete` action에서 cur가 장착 중이었다면 캡 처리를 한 트랜잭션 안에서 묶거나, `replaceItem(slot, newId, oldId)` 같은 단일 트랜잭션 action 추가
  - (B) 클라이언트: `handleEquipFromStash`도 `handleReplace`와 같은 순서(equip new → delete old)로 변경. 단, 이 경우 같은 슬롯에 두 아이템이 잠시 unequipped로 남는 점은 `equipItem` batch가 자동 처리하므로 안전
- 비고: 이는 캐릭터·스탯 모듈의 [HIGH 3] (last_regen_at 미갱신)과 같은 PATCH 경로 위에서 발생하는 별개 이슈. 회복 시계와는 무관한 캡 손실.

### [HIGH] 3. 옵션 SoT(텍스트) ↔ `battle.ts` 파서 정합 강제 수단 없음

- 위치: `lib/db/seed.ts:195-219` (`seedItemAbilityPool`), `lib/battle.ts:336-350` (옵션→스탯 분기)
- 현상: 옵션 한국어 이름이 곧 SoT. battle.ts의 파서는 if/else 체인으로 정확 일치를 요구. seed에 새 ability를 넣으면 battle.ts에 분기를 추가해야 하지만, 자동 검출 수단이 없다.
  - 현재 `item_ability_pool`의 15개 이름과 `battle.ts:336-350`의 분기 키 비교 결과 모두 매칭되지만, 다음과 같은 함정이 잠재:
    - "방어력"과 "물리방어력" 두 키가 같은 변수(ePdef)에 들어감 (`battle.ts:338`). seed에 "물리방어력"이라는 ability는 없고 "방어력"만 존재 — `||` 분기가 dead branch. 향후 seed에 "물리방어력"이 추가되면 자동으로 ePdef 가산되지만, 슬롯 메인이 "물리방어력"인 경우(`item_slot_table`의 helmet/armor/belt가 "방어력", glove의 excluded에 "물리방어력"이 포함됨)와의 의미 정합성이 모호.
  - `item_slot_table`의 `excluded`에 "물리방어력"이 들어있는데(`seed.ts:182,184,185`), `item_ability_pool`에는 "물리방어력"이라는 이름의 능력이 없으므로 excluded 매칭이 dead. 즉 "물리방어력"은 schema/seed 양쪽에 사용되었지만 ability pool에는 없음 → 해당 슬롯의 excluded가 일부 무의미.
- 시나리오: 새 능력 "치명타저항" 추가 → seed에 INSERT, battle.ts 파서에 분기 추가 잊음 → 가챠는 옵션에 적지만 전투에서 silent ignore
- 제안:
  - (A) 빌드 타임 lint: `lib/battle.ts`의 분기 키와 `item_ability_pool.name`을 비교하는 스크립트를 `package.json`의 `prebuild`에 추가
  - (B) 런타임: `buildPlayerCombatStats` 안에 "알 수 없는 키"를 logger로 흘려서 운영 중 발견 가능하게
  - (C) excluded와 ability pool의 이름 정합성도 시드 시점에 sanity check
- "물리방어력"이 ability pool에 없는 점은 별도 데이터 정합성 항목으로도 남김 — slot_table의 excluded 데이터를 한 번 정리할 필요.

### [MEDIUM] 4. `unit="HP"` / `unit="MP"`인 ability의 `base_value`가 무시됨

- 위치: `lib/db/seed.ts:210-211` (HP증가/MP증가), `lib/game/gacha.ts:127-135` (`rollAbilityValue`)
- 현상: `rollAbilityValue`는 `unit === "%"`만 분기하고 그 외(Pt, HP, MP, 미정의 단위)는 모두 동일 분기로 흘러간다. seed의 HP증가(base_value=50)와 MP증가(base_value=30)는 분기 진입 시 grade의 stat_min/stat_max만 사용하고 base_value는 사용 안 됨.
- 영향:
  - 결과적으로 HP증가가 grade에 따라 stat_min*0.3*ratio ~ stat_max*0.5*ratio 정수값으로 굴려짐. 예: C 등급 HP증가 → `floor(5*0.3*ratio)` ~ `floor(10*0.5*ratio)`. SUB 슬롯의 ratio=0.5라면 0~2 정도. 의도된 "최대HP +50" 시드값과 실측 결과의 괴리가 큼
  - SSR 등급에서도 stat_min*0.3*0.5 ~ stat_max*0.5*0.5 = 15~42 정도. seed의 base_value=50과 다름
- 원인: `rollAbilityValue`가 unit을 "%"와 그 외 두 갈래로만 분기. HP/MP unit의 의도(고정 base_value 기반 롤링)가 구현 안 됨
- 시나리오: 신규 ability "체력회복"을 base_value=80, unit="HP"로 추가 → 가챠 결과는 grade의 stat 범위 기반으로 굴려져 base_value와 무관한 값 출력
- 제안: 명시적 분기 추가 — `unit === "HP" || unit === "MP"`인 경우 `formatOpt(name, base_value * ratio * (1 + Math.random()), "")` 또는 의도된 산식. 정책 결정 필요. 또는 unit을 `%`/`Pt` 두 가지로만 강제하고 HP/MP 단위 자체를 폐기.
- 비고: 옵션 텍스트는 "HP증가 +N" 형태(단위 없음, formatOpt가 unit !== "%" → Math.round)로 출력되므로 battle.ts의 `eHp` 가산은 정상 동작. 즉 동작은 하되 값의 분포가 의도와 다름 (밸런스 이슈가 아니라 의도 미구현).

### [MEDIUM] 5. `item_passive_to_skill_v1` 마이그레이션이 `item_passive_pool`을 정리하지 않음

- 위치: `lib/db/schema.ts:622-627`, `lib/db/seed.ts:221-234`
- 현상: 마이그레이션은 `equipment.options` 텍스트의 옛 이름 4개를 새 스킬 이름으로 REPLACE만 한다. `seedItemPassivePool`이 시드한 row 4개(더블어택/생명흡수/방어무시/반사)는 그대로 잔존. 그러나 #1과 같이 가챠는 `skill_table`에서 풀을 가져오므로 잔존 데이터가 가챠에는 영향 없음.
- 영향: 데이터 잔여물. 향후 누군가 `item_passive_pool`을 직접 SELECT해서 UI에 표시하거나 쓸 경우 옛 이름이 노출되어 일관성 문제 발생
- 제안: #1 처리 시 함께 — `item_passive_pool` 테이블 자체를 deprecate하거나 마이그레이션으로 정리

### [MEDIUM] 6. `discardGacha` 액션과 가챠 결과 폐기 흐름의 SoT 일관성

- 위치: `app/api/inventory/route.ts:48-52`, `components/game/ItemsTab.tsx:106-111`
- 현상: 가챠 결과 폐기는 두 경로:
  - (a) `handleDiscard`: `pendingReplace`가 있을 때 (현재 장착 중인 슬롯에 가챠 결과가 나옴) → `discardGacha` 액션으로 pity_count 증가
  - (b) 자동 장착 분기(`handleGacha:81-85`): `currentEquipped`가 없으면 그냥 equip → pity_count 0으로 리셋
  - 한편 `handleEquipFromStash`나 일반 보관함의 `handleDelete`는 `delete` 액션만 호출. pity 카운트와 무관
- 영향: pity 정책은 "가챠 직후 결과를 폐기"한 경우만 카운트 증가하는 의도로 보임. 그러나 사용자가 "현재 장착 중이라 자동장착 안 되는 가챠 결과"를 한참 뒤에 보관함에서 삭제하면 — 일반 delete가 호출되어 pity 미적립. 즉 결과를 즉시 결정하지 않고 미루면 pity 손실
  - 또한 `pendingReplace`가 떠 있는 동안 다른 페이지로 이동해 state 잃으면 pity 적립 기회를 잃음
- 시나리오: 가챠 → currentEquipped가 있어서 `pendingReplace` 모달 표시 → 사용자가 모달을 닫지 않고 다른 탭(BattleTab) 이동 → 돌아왔을 때 `pendingReplace`는 사라져 있고 가챠 아이템은 보관함에 unequipped 상태로 존재 → 일반 삭제 시 pity 미적립
- 원인: pity 카운트가 "가챠 의도 충족 여부"를 결정하는 정책인데, 가챠 결과의 라이프사이클(보관/장착/폐기)에서 시점이 분산
- 제안: 두 가지 중 결정.
  - (A) `delete` 액션도 "최근 가챠 결과인지"(예: created_at이 N분 이내인지) 체크해 pity 카운트 — 복잡
  - (B) 가챠 결과를 항상 즉시 처리 강제 (모달이 떠 있는 동안 다른 탭 이동 차단) — UX 부담
  - (C) 현 정책 유지하되 "결정 미루면 pity 미적립" 명시 + 모달 dismiss 시 `discardGacha` 자동 호출

### [MEDIUM] 7. 가챠 트랜잭션 외부에서 `pity_count` 갱신 — race 가능

- 위치: `app/api/inventory/route.ts:42-52, 84-119`
- 현상:
  - PATCH의 `equip` 분기: `equipItem(itemId)` 후 `updateCharacter({ pity_count: 0 })` — 두 호출이 별개. 도중 실패 시 pity 미리셋 + 장착 적용
  - PATCH의 `discardGacha` 분기: `deleteEquipment` → `getCharacter` → `updateCharacter({ pity_count: 현재값+1 })` — read-modify-write가 트랜잭션 밖. 동시 두 요청 시 pity 증가 1회 누락 가능
  - POST(가챠): pity 적용 후 등급 결정은 `rollGachaItems` 안에서 행해지지만 pity_count는 함수 호출 직후 사용된 값으로 고정. 트랜잭션 안에서 ticket 차감만 하고 pity는 PATCH 이후에야 변동 → 한 번의 가챠 안에서는 stale 위험은 없음
- 영향: 단일 사용자 환경에서는 동시 요청이 거의 없어 실 영향 낮음. 다중 가챠/장착 빠르게 누르는 시나리오에서 pity off-by-one 가능
- 제안: equip + pity_count 리셋, discardGacha의 read-modify-write를 `tx`로 묶거나 `UPDATE character SET pity_count = pity_count + 1 WHERE id=1`로 SQL 자체에서 atomic하게 처리

### [MEDIUM] 8. `pickGrade` 폴백(`weight ≤ 0`) 시 균등 분포로 떨어지는 위험

- 위치: `lib/game/gacha.ts:76-86`
- 현상: 모든 grade의 weight 합이 0 이하면 `adjusted[Math.floor(Math.random() * adjusted.length)]`로 균등 폴백. 운영자가 모든 weight를 잘못 0으로 설정하면 1/7 확률로 UR이 나오게 됨 — 정상 가챠 분포(0.1%)와 700배 차이.
- 영향: 운영 실수 방어가 약함. UI(`/api/items/grades` 등)로 weight 변경이 노출되어 있다면 위험. `updateItemGradeWeight` 함수가 존재(`items.ts:9`)하므로 운영 가능성 있음.
- 시나리오: 어드민 UI로 모든 weight=0 입력 → 운영자가 의도는 "가챠 막기"였는데 실제로는 균등 가챠

### [LOW] 11. `applyItemPassives`와 옵션 정렬 — 매칭 검증 자동화 부재

- 위치: `lib/battle.ts:48-59`
- 현상: `[스킬명]` 옵션을 `skill_table.name` 정확 일치로 invested+1. 매칭 실패 시 silent — counts에 잡혀도 skills.map의 find가 없어서 그냥 사라짐.
- 자세히: 코드는 `skills.map(s => ({ ...s, invested: s.invested + (counts[s.name] ?? 0) }))` 형태. counts에 skill_table.name과 매칭 안 되는 키가 들어가면 그냥 무시됨 — `[연속공격]`이 옛날 이름 `[더블어택]`으로 남아있는 옵션이 있다면 무발동 (마이그레이션 `item_passive_to_skill_v1`이 처리하지만 그 이전 옵션이 다른 형식이라면 누락 가능).
- 제안: `Object.keys(counts)`와 `skills.map(s=>s.name)` 차집합을 dev 모드에서 console.warn — 운영 발견 장치 마련.

### [LOW] 12. `excluded` 매칭이 main_ability와 결합되면서 dead-branch 가능성

- 위치: `lib/game/gacha.ts:175-194`
- 현상:
  - 서브(BaseStat) 분기 (`:175-182`): `category === "BaseStat" && !excluded.has(a.name) && !used.has(a.name)`
  - 전투(Combat) 분기 (`:186-194`): 위에 더해 `&& a.name !== slot.main_ability`
  - 주의: BaseStat 분기에는 `slot.main_ability` 제외가 없다. 하지만 BaseStat 항목들("STR(힘)" 등)은 어느 슬롯의 main_ability도 아니므로 dead-safe. 단, 향후 main_ability를 STR/VIT 같은 BaseStat으로 잡는 슬롯을 추가하면 BaseStat 서브에서 main과 중복 가능.
  - excluded는 string 배열인데 비교는 `a.name`으로 한다. seed의 excluded에 "물리방어력"이 있으나 (#3) ability pool에는 "물리방어력" 이름의 row가 없으므로 dead 비교가 매번 일어남. `item_slot_table`의 excluded 데이터 정합성 점검 필요.
- 제안: 슬롯 main_ability 카테고리가 BaseStat인 경우를 대비해 BaseStat 분기에도 `a.name !== slot.main_ability` 추가 (방어 코드). excluded 데이터 정리는 별도.
