## 요약

- 발견된 이슈: **blocker 0 / high 3 / medium 4 / low 4**
- 핵심 권고
  1. `last_regen_at` 갱신 누락이 3곳 — 수동 HP 편집/레벨업 풀회복/장비 변경 후 자연회복이 의도와 다르게 계산됨

## 발견사항

### [HIGH] 1. `last_regen_at` 갱신 누락 — `/api/character` PUT

- 위치: `app/api/character/route.ts:107-157` (`PUT` 핸들러)
- 현상: PUT은 `current_hp` / `current_mp`를 수동 변경할 수 있는데, 그 결과를 DB에 쓸 때 `last_regen_at`은 함께 갱신되지 않는다.
- 시나리오:
  1. SettingsDrawer에서 `current_hp`를 100 → 110 으로 수동 편집 → PUT
  2. 1시간 후 페이지 로드 시 `calcRegen(current=110, max=110, ..., lastRegenAt=어제 시각)` 호출
  3. `current >= max`라 early return → 표면상 문제 없음
  4. 그런데 그 사이 전투에서 HP가 80으로 감소 → 다음 calcRegen은 `elapsed = (어제 → 지금)` 기준으로 회복 계산 → 한 번에 풀회복
- 원인: 전투 라우트(`/api/battle`)는 `last_regen_at = now()`로 매번 갱신하지만, PUT 라우트는 갱신하지 않아 두 라우트가 비대칭
- 제안: `fields`에 `current_hp` 또는 `current_mp`가 포함된 경우 `last_regen_at = now()`도 함께 update. 안전 측면에서 매 PUT 호출마다 갱신해도 큰 손해 없음 (PUT은 빈도 낮음)

### [HIGH] 2. `last_regen_at` 갱신 누락 — `gainExp` 레벨업 풀회복

- 위치: `lib/game.ts:94-102`
- 현상: 레벨업 시 `updates.current_hp = Math.round(cs.max_hp)` 로 풀회복하지만 `last_regen_at`은 업데이트 객체에 없음.
- 시나리오:
  1. 레벨업으로 HP 풀회복 (current=max)
  2. 전투 입장 → 전투에서 HP 50% 소비 → 전투 라우트가 `last_regen_at = now()` 갱신해서 회복 (이 케이스는 보호됨)
  3. 그러나 레벨업 후 전투 외 다른 경로(예: SettingsDrawer로 current_hp 수동 감소, 또는 향후 HP 소비 액션 추가)로 HP가 줄어든 직후 `page.tsx`의 calcRegen이 호출되면 `last_regen_at`이 레벨업 이전 시각이라 비정상 회복
- 원인: 풀회복과 last_regen_at 갱신이 한 트랜잭션에서 묶이지 않음
- 제안: `gainExp`의 `leveledUp` 분기 안에 `updates.last_regen_at = now()` 추가 (이미 `updated_at = now()`가 있으므로 같은 값 사용 가능)

### [HIGH] 3. `last_regen_at` 갱신 누락 — `/api/inventory` PATCH

- 위치: `app/api/inventory/route.ts:65-71`
- 현상: 장비 해제로 max_hp가 감소했을 때 `current_hp`를 새 max로 캡하지만 `last_regen_at`은 갱신하지 않음
- 시나리오:
  1. HP 50/100 상태에서 1시간 경과
  2. 장비 해제 → 새 max 80 → current 50으로 유지 (cap 안 걸림)
  3. `last_regen_at`은 변경 안 됨
  4. 직후 `page.tsx`의 calcRegen 호출 → elapsed = 1시간 → max 80 캡으로 풀회복으로 보임 (회복 속도 의도와 다름)
- 원인: 장비 변경 시점에서 HP 시계 리셋이 의도된 정책인지 명시되지 않음. battle 라우트는 매번 리셋함
- 제안: 두 가지 중 선택. (a) PATCH에서 `last_regen_at = now()` 갱신 → 장비 변경을 회복 기준점으로 취급. (b) "장비 변경은 회복 기준점이 아니다"라는 정책을 명시 → 현재 동작 유지하되 주석 추가. (a) 권장.

### [MEDIUM] 4. `recalcHpMp` 호출 누락 — 장비 변경

- 위치: `app/api/inventory/route.ts:39-73` (PATCH)
- 현상: 장비 장착/해제 후 `character.max_hp` / `max_mp` DB 컬럼이 갱신되지 않음. effective max는 `buildPlayerCombatStats`로 매번 계산되어 정확하지만, character row의 stored max_hp/mp는 base 기준으로만 유효
- 영향:
  - `/api/character` GET: effective를 별도 반환하므로 사용자에게는 정상 표시
  - `lib/battle.ts` 등 다른 코드: `char.max_hp` 직접 참조하는 곳은 grep 결과 없음 → 현 시점 사용자 가시 영향 없음
  - 잠재 위험: 신규 코드가 `char.max_hp`를 직접 사용하면 base 값(아이템/스킬 미포함)을 받게 됨
- 원인: `character.max_hp`의 의미가 "base only"인지 "effective"인지 코드 전반에 명시 안 됨. gainExp는 base로 저장하고, 다른 라우트는 effective를 별도 계산
- 제안:
  - 정책 명문화: "DB `character.max_hp`는 base + VIT 기반 max다. effective max는 항상 buildPlayerCombatStats로 계산하라"는 주석을 schema.ts 또는 character query 파일에 추가
  - 또는 PATCH에서 recalcHpMp(현재 stat 기준)도 호출해 항상 정합 유지

### [MEDIUM] 5. `recalcHpMp` 호출 누락 — 스킬 패시브 변경

- 위치: `app/api/skills/route.ts` (스킬 invested 변경 PUT)
- 현상: HP_PCT 패시브 변경 시 effective max는 변경되지만 `character.max_hp` DB 값은 그대로
- 영향: #4와 동일. 사용자 가시 영향은 거의 없지만 일관성 결여
- 제안: #4 정책을 적용하면 자연히 해소

### [MEDIUM] 6. `gainExp` 트랜잭션 내 equipment read의 동시성

- 위치: `lib/game.ts:95-99`
- 현상: 트랜잭션 내 `SELECT is_equipped, options FROM equipment` 실행. 동시에 다른 요청이 inventory PATCH로 equipment를 변경하면 isolation level에 따라 stale read 가능
- 영향: 단일 사용자 앱이라 실제 동시 요청 가능성 낮음. Turso libsql write 트랜잭션은 SQLite와 유사하게 직렬화되므로 현 환경에서는 안전한 편
- 제안: 현 단계에서는 코드 수정 불필요. 멀티유저 확장 시 재검토 필요 사항으로 명시
- 후속: `lib/game.ts:42-94` 주석의 "트랜잭션 안에서 재read해 stale max_hp 방지"가 정확히 이를 다루고 있음 → 의도된 디자인 확인됨 ✓

### [MEDIUM] 7. `/api/character` DELETE의 부분 초기화 모호성

- 위치: `app/api/character/route.ts:68-102`
- 현상: `mode=full`이라도 `character` 행의 level, total_exp, str/vit/dex/int/luk, draw_tickets, last_regen_at 등은 리셋 안 됨. 로그류와 equipment, skill_log, attendance_log, pending_battle_monster만 정리
- 영향: "캐릭터 완전 초기화"라는 이름과 실제 동작이 어긋남. 사용자가 stat을 재분배 못 한 채 항목만 비우는 결과
- 제안: 두 가지 중 명확화. (a) mode='full'에 character의 stat/level/exp/tickets/last_regen_at도 초기값으로 리셋하는 분기 추가. (b) 함수명/주석에 "데이터 로그만 초기화. 캐릭터 능력치는 별도 PUT으로"라고 명시
- 별도 주의: `last_regen_at`이 리셋되지 않은 채 새 항목이 만들어지면, 새 항목 첫 calcRegen 호출 시 과거 시각 기준 회복으로 풀회복 계산됨. mode='full' 시 `last_regen_at = now()` 갱신 권장

### [LOW] 8. effective max 계산 코드 4중복

- 위치: `app/api/character/route.ts:21-29` (GET), `:138-143` (PUT), `app/api/inventory/route.ts:55-64` (PATCH), `app/api/battle/route.ts:51-63` (POST)
- 현상: 모두 다음 패턴 반복:
  ```
  const [bcfg, equipment, allSkills] = await Promise.all([...])
  const equippedOptions = equipment.filter(e => e.is_equipped===1).map(e => e.options)
  const boostedSkills = applyItemPassives(allSkills, equippedOptions)  // 일부만
  const cs = buildPlayerCombatStats(char, equippedOptions, bcfg, boostedSkills 또는 allSkills)
  ```
- 일관성 차이: PATCH(inventory)와 battle은 `boostedSkills`(applyItemPassives 거친 것) 사용, GET/PUT(character)와 game.ts gainExp는 `allSkills`(applyItemPassives 미적용) 사용 → **장비 옵션의 `[스킬명]` 패시브가 캐릭터 GET/PUT/gainExp에서는 무시됨**
- 영향:
  - `/api/character` GET의 effective는 장비 스킬 패시브를 반영하지 않은 값을 사용자에게 표시 → UI 불일치 가능 (전투에서는 적용되는데 표시는 미적용)
  - `gainExp` 레벨업 풀회복 시 장비 스킬 패시브 미적용 max로 풀회복 → 직후 전투 입장 시 max가 더 크게 계산되어 풀회복 미달
- 제안: helper 함수로 추출 + boostedSkills 사용으로 통일
  ```ts
  // lib/character/stats.ts 신규
  export async function getEffectiveStats(char) {
    const [bcfg, equipment, allSkills] = await Promise.all([...])
    const equippedOptions = equipment.filter(...).map(...)
    const boostedSkills = applyItemPassives(allSkills, equippedOptions)
    return buildPlayerCombatStats(char, equippedOptions, bcfg, boostedSkills)
  }
  ```
- **이 항목은 LOW로 분류했지만 일관성 문제를 동반하므로 실제 영향은 MEDIUM에 가까움. 별도 추적 권장**
