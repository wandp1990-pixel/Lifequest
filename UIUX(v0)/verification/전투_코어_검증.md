## 발견사항

### [HIGH] 1. `restoreMode="half"` 의 이중/혼합 회복 — startHp 회복 + 결과 후 half 회복

- 위치: `app/api/battle/route.ts:78-99`
- 현상:
  - `restoreMode="half"`일 때 (`route.ts:88-89`) `startHp = regenedHp`, `startMp = regenedMp`로 자연회복된 값으로 전투 시작
  - 전투 후 (`:94-99`) `finalHp = min(max, result.player_final_hp + max/2)`로 **추가** half 회복
  - 즉 "half"는 사실상 "자연회복으로 시작 → 전투 → 끝났을 때 max/2만큼 더 추가" 가 됨. seed 라벨 "half: 절반 회복"의 의도와 어긋남 (절반 회복이 시작 시점인지 종료 시점인지 모호)
- 원인: regen 적용 분기와 restoreMode 후처리 분기가 같은 함수에 같은 변수("`startHp`"는 regen, "`finalHp`"는 restoreMode 후처리)로 분리되어 있어, 두 정책이 의도치 않게 합산됨
- 시나리오:
  1. max_hp=200, current=10, last_regen_at=1시간 전 → regenedHp ≈ 200(거의 풀회복)
  2. 전투 시작 시 HP 200으로 진입, 100까지 깎인 채 패배
  3. finalHp = min(200, 100 + 100) = 200 → 사실상 풀회복으로 보임
  4. 같은 시나리오에서 last_regen_at=방금 전이면 regenedHp=10 → 전투 시작 HP 10, 0까지 깎이고 패배 → finalHp = min(200, 0 + 100) = 100
  5. **last_regen_at 시각에 따라 동일 패배 결과의 finalHp가 200 vs 100 으로 갈림**
- 제안:
  - 정책 결정: half 모드의 의미를 (a) "전투 시작 시 50% 보장" (b) "전투 종료 시점 + 50% 보너스" 중 하나로 명확화
  - (a) 채택 시 `restoreMode==="half"`이면 `startHp = max(regenedHp, max/2)`, finalHp 후처리 제거
  - (b) 채택 시 startHp는 regenedHp 그대로, finalHp 후처리 유지하되 라벨을 "전투 후 50% 추가 회복"으로 수정
  - 어느 쪽이든 `none` 모드는 startHp=regenedHp, finalHp=result.player_final_hp 로 단순화 (현재도 이렇긴 함)

### [HIGH] 2. `restoreMode="none"`/시간초과 시 `last_regen_at` 갱신 정책의 의도치 않은 회복 시계 리셋

- 위치: `app/api/battle/route.ts:101, 121, 137`
- 현상: 승/패 양쪽 분기에서 `last_regen_at = regenAt = now()`로 무조건 갱신. `restoreMode="none"` 정책("HP 회복 없음")인데 last_regen_at이 리셋되므로, 다음 페이지 로드 시 `calcRegen`이 "방금부터 회복 시작" 으로 동작. 사용자 입장에서 "회복 안 됨"이라는 정책과 어긋남
- 시나리오 (none 모드):
  1. HP 100/100, last_regen_at = 60분 전, 60분 동안 안 돌아감
  2. 전투 입장 → 자연회복 적용으로 startHp=100 → 전투에서 50까지 떨어져 패배
  3. finalHp=50, last_regen_at=now() 갱신
  4. 다음 페이지 로드 시 calcRegen(50, 100, ...) → 즉시 회복 시작
  5. 만약 last_regen_at을 갱신하지 않았다면 60분 + 다음 idle 시간이 누적되어 더 빨리 풀회복됨 — 즉 현재 구현은 사용자에게 **불리한** 회복
- 원인: 회복 시계의 기준이 "마지막으로 HP가 변동한 시각"인지, "마지막으로 회복 계산이 commit된 시각"인지 코드에 명시 안 됨. 라우트마다 다른 정책
- 제안:
  - 정책 결정 후 주석으로 명문화: "전투 시작 시 자연회복 1회 적용, 종료 시 last_regen_at=now() 로 리셋. 즉 회복 시계는 전투마다 리셋됨"
  - 또는 restoreMode="none" 분기에서 last_regen_at을 갱신하지 않고 char.last_regen_at 유지 (불공평 회복 방지)

### [MEDIUM] 5. `regenedMp` 계산 시 패시브/장비 미반영 stat 사용

- 위치: `app/api/battle/route.ts:84-85`
- 현상:
  - `regenedHp = calcRegen(char.current_hp, effMaxHp, char.vit + itemBonus.vit, char.last_regen_at)`
  - `regenedMp = calcRegen(char.current_mp, effMaxMp, char.int_stat + itemBonus.int_stat, char.last_regen_at)`
  - effMaxHp/Mp는 `playerCombat.max_hp`(스킬 패시브 HP_PCT 포함)인데, 회복속도를 좌우하는 stat은 **char.vit + itemBonus.vit**(스킬 패시브 미반영)
- 영향: `PASSIVE_HP_01`(HP_PCT) 같은 스킬 패시브가 max_hp에는 반영되지만, 회복 속도 계산식 `Math.floor(max * 0.10) + Math.floor(stat / 10)`의 stat 항에는 반영 안 됨 → 회복 속도가 effective stat 대비 낮게 계산
  - 단, calcRegen 첫 항이 `max * 0.10`으로 max에 의존하므로, 패시브 HP_PCT 적용된 max로는 회복량 자체는 늘어남. 둘째 항만 stat-기반 보너스
- 원인: VIT_PCT/INT_PCT 계열 패시브는 현재 seed에 없음. DEX_FLAT/LUK_FLAT는 있지만 회복엔 무관. 따라서 현 시점 사용자 영향은 작지만 "패시브 보너스가 max에는 반영, stat에는 반영 안 됨" 이라는 비대칭이 코드 로직 상 잠재 위험
- 제안: 명시적 정책 — calcRegen은 base stat 기준임을 주석으로 명시. 또는 `playerCombat`이 `vit_total`/`int_total`을 노출하도록 확장 후 그 값 사용

### [MEDIUM] 7. `pending_battle_monster` JSON에 저장 시점의 stale 위험

- 위치: `app/api/battle/route.ts:138`, `BattleTab.tsx:33-44`
- 현상: 패배/시간초과 시 `JSON.stringify(monster)` 저장. 그 후 사용자가 레벨업/장비 교체/스킬 투자 → 강도가 달라진 상태에서 같은 몬스터로 재도전. **monster 객체는 그대로**(생성 당시의 stats/coeff 포함)이지만 player 측은 강해짐 → 의도된 디자인일 가능성 (재도전 시 더 강한 캐릭터로 같은 적을 잡으라는 컨셉)
- 그러나 monster.total_coeff는 생성 당시의 `clearCount`/`playerLevel`로 계산된 값이라 **레벨업 후 재도전 시 같은 monster 재사용**이 곧 "현재 레벨에 비해 약화된 적"이 됨. 이게 의도인지 확인 필요
- 또한 사용자가 max_cleared_grade를 변경(다른 경로?)할 일은 없지만, BattleTab.tsx:38-40에서 `maxClearedIdx + 1`보다 grade가 높은 saved monster는 표시 안 함 (admin이 max_cleared_grade를 강제로 낮춘 후 stale pending이 남으면 표시 안 됨 — DB는 그대로 유지)
- 제안:
  - 의도라면 주석으로 명시 ("재도전은 패배 시점의 몬스터 그대로. 사용자 강화 후 잡는 것이 컨셉")
  - DB의 stale pending 정리 정책 추가 (예: 24시간 이상 미도전 pending은 자동 clear)

### [MEDIUM] 8. `battle_log` 10개 cap 동시성 — DELETE 서브쿼리의 race

- 위치: `app/api/battle/route.ts:158-160`
- 현상: 트랜잭션 안에서 `INSERT INTO battle_log` → `DELETE FROM battle_log WHERE id NOT IN (SELECT id FROM battle_log ORDER BY id DESC LIMIT 10)` 순서로 실행. 같은 트랜잭션 내라 INSERT가 보이는 상태에서 DELETE 수행 → 항상 최신 10개 유지 ✓
- 영향: libsql/SQLite의 write 트랜잭션은 직렬화되므로 단일 사용자 환경에서 11개 이상 남을 가능성 거의 없음. 동시성 위험 낮음
- 그러나: 같은 cap 로직이 `lib/db/queries/battle.ts:3-15` `addBattleLog()`에도 있는데, 이 함수는 라우트에서 사용되지 않음 (`grep` 결과 호출처 없음). **죽은 함수**
- 제안: `addBattleLog` 미사용 함수 제거 또는 라우트의 인라인 SQL을 이 함수로 추출. 두 곳에 같은 SQL이 흩어진 상태가 더 위험

### [MEDIUM] 9. battle-config PUT의 silent UPDATE — 존재하지 않는 key INSERT 안 됨

- 위치: `app/api/battle-config/route.ts:18-23`, `lib/db/queries/config.ts:35-41`
- 현상: `updateBattleConfigValue` 가 UPDATE만 수행. battle_config에 없는 새 key를 PUT하면 "ok: true" 응답하지만 DB는 변경 없음 (silent noop)
- 영향: 키 오타 시 UI가 변경 반영 안 되는데 에러도 없음. 단일 사용자 / 개발자 본인 사용이라 큰 위험은 아니지만 디버깅 시간 낭비
- 또한 PUT은 인증/권한 체크 없음 — 외부에서 임의 호출 가능. 단일 사용자 앱이지만 production 배포된 endpoint이므로 누구나 `restore_hp_after_battle="none"` 같은 변경 시도 가능
- 제안:
  - UPDATE 대신 `UPDATE ... ` + `if (changes === 0) badRequest("unknown key")` 분기 추가 (libsql는 changes() 함수 또는 `result.rowsAffected` 사용 가능)
  - production endpoint에 최소한의 토큰 검증 권장 (별도 모듈)

### [LOW] 14. `regenAt = now()` 와 `result` 라우트의 transaction 외부 read — char 객체 stale

- 위치: `app/api/battle/route.ts:51-66`, tx 내부 `:105-141`
- 현상: `char`/`equipment`/`allSkills`를 트랜잭션 외부에서 읽음. tx 내부 `tx.execute(UPDATE character SET draw_tickets = ? WHERE id = 1, args=[char.draw_tickets + reward])`. 읽은 시점과 UPDATE 시점 사이에 다른 요청이 character row를 변경하면 lost update 위험 (예: 동시에 EXP 활동으로 stat 분배가 일어나면 stat_points 등 다른 컬럼은 OK, draw_tickets는 race)
- 영향: 단일 사용자라 동시 요청 가능성 매우 낮음. libsql write tx serializable이라 안전
- 제안: 현 단계 수정 불필요. 멀티유저 확장 시 트랜잭션 안에서 SELECT 후 UPDATE, 또는 `UPDATE character SET draw_tickets = draw_tickets + ?` 패턴으로 변경 권장

### [LOW] 15. `getActiveSkills` 호출 후 boostedSkills로 재호출되어 invested 가산이 active에도 적용

- 위치: `app/api/battle/route.ts:61-63`, `lib/battle.ts:42-44`
- 현상: `boostedSkills = applyItemPassives(allSkills, equippedOptions)` → `activeSkills = getActiveSkills(boostedSkills)`. applyItemPassives는 type 무관하게 모든 skill의 invested에 counts 가산. 즉 장비 옵션의 `[연속공격]` 같은 액티브 스킬 이름이 박히면 **액티브 스킬 invested+1** 효과
- 영향: 의도된 디자인일 가능성 있음 (장비로 액티브 스킬도 강화). 그러나 명문화 안 됨. seed의 item_passive_pool은 "더블어택/생명흡수/방어무시/반사" 4개 (모두 미작동, [LOW 13] 참조)이므로 현재 silent fail로 영향 없음
- 제안: 의도 확인 후 주석 추가. 패시브 스킬만 강화 의도라면 `applyItemPassives` 내에서 type==="passive"만 가산하도록 필터 추가
