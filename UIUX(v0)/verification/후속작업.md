## 1. ADMIN_SECRET env 등록 + UI 헤더 전송

**현재 상태**: `/api/battle-config` PUT 과 `/api/skill-db` POST/PUT/DELETE 가 fail-closed 통일됨 (env 미설정 시 503). production 에 `ADMIN_SECRET` 미설정이라 두 라우트의 쓰기가 모두 차단된 상태. UI 측 `SkillDbPanel` / battle-config 패널은 헤더를 보내지 않아, env 를 등록하는 순간 패널 쓰기가 즉시 깨진다.

**작업**:
1. `ADMIN_SECRET` 값 결정 (랜덤 문자열 OK)
2. `npx vercel env add ADMIN_SECRET production` (preview/dev 동일 값 또는 별도)
3. UI 헤더 주입 방식 결정:
   - **A. SettingsDrawer 에 admin 토큰 입력란** — localStorage 저장, fetch 시 `Authorization: Bearer <token>` 헤더 추가
   - B. server-action 경유 (변경 큼)
   - C. `NEXT_PUBLIC_ADMIN_SECRET` 노출 — **권장 안 함**
4. 패널 측 fetch helper 수정. fetch 실패 시 토스트로 "관리자 토큰 확인" 안내

**파일**: `components/game/SettingsDrawer.tsx`, `components/game/SkillDbPanel.tsx`, battle-config 패널.

## 2. runBattle 분해 + trigger_condition enum 화

**배경**: `lib/battle.ts:runBattle` 한 함수에 데미지 계산, 명중/회피, 치명타, 스킬 트리거 7~8개 분기, HP/MP 관리, 로그 누적이 모두 인라인 (200줄+). `trigger_condition` 은 한국어 문자열 비교 (`"HP 25% 이하"`, `"매 3턴"` 등) 가 7곳에 흩어져 있어 한 글자 오타 시 silent fail.

**작업**:
1. `lib/skills/triggers.ts` 신규 — trigger_condition 상수
   ```ts
   export const SKILL_TRIGGERS = {
     BATTLE_START:  "전투시작",
     FIRST_STRIKE:  "선공",
     EVERY_3_TURNS: "매 3턴",
     ON_HIT:        "명중시",
     ON_CRIT:       "치명타시",
     ON_EVADE:      "회피시",
     ON_DEATH:      "사망시",
   } as const
   ```
2. `runBattle` 을 분해:
   - `calcDamage(attacker, defender, isSkill)` — 데미지 + 치명타/명중/회피 판정
   - `applyTriggeredSkills(phase, ctx)` — 트리거 시점별 효과 dispatcher
   - `runTurn(player, monster, ctx)` — 한 턴 처리
   - `runBattle` 은 상위 loop 만 유지
3. `seed.ts` 의 trigger_condition 값을 `SKILL_TRIGGERS` 상수 참조로 교체 (런타임 시드 변경 시 마이그레이션 필요 여부 검토)

**사전 조건 (회귀 방지)**: 변경 전 fixture 10~20개 캡처 + 변경 전후 결과 동일성 비교 스크립트.
- 평타 명중 (STR 30, VIT 10 vs C급 슬라임)
- 평타 회피 (DEX 격차 큰 경우)
- 치명타 발생 (LUK 50 + 치명타 패시브)
- HP 25% 트리거 "역전"
- 매 3턴 트리거 (9턴 전투에서 3회 발동)
- 사망 시 SURVIVE
- 명중 시 EXTRA_HIT
- 패배 후 재도전 (`pending_battle_monster` 동일성)
- `restoreMode=half` 시작 HP 보장
- `restoreMode=none` `last_regen_at` 시계 유지

테스트 위치 후보: `lib/__tests__/battle.test.ts` (vitest 도입 동반).

**파일**: `lib/battle.ts`, `lib/skills/triggers.ts` (신규), `lib/db/seed.ts`. 가능하면 `lib/battle/` 디렉토리 분리 후 `lib/battle.ts` 를 re-export shim 유지.

## 3. getPassivePool SoT 정리 + item_passive_pool seed 재작성

**배경**: `lib/db/queries/items.ts:getPassivePool` 이 함수명과 다르게 `skill_table` 전체를 읽어 액티브 스킬도 가챠 패시브 풀에 포함시킴. `item_passive_pool` seed 의 4개 row (더블어택/생명흡수/방어무시/반사) 는 어떤 코드도 읽지 않는 dead. 그 4개 패시브는 `attack()` 내 변수가 0이라 사실상 비활성 (`lib/battle.ts:308-309`). 현재는 임시 경고 주석만 추가됨.

**두 갈래 정책 결정 필요**:
- **A. 가챠 패시브 풀 SoT = `skill_table`**
  - `getPassivePool` → `getGachaPassiveSource` rename
  - `item_passive_pool` seed/마이그레이션 row 4개 제거 또는 `is_active=0`
  - 가챠 패시브 = 보유 스킬 강화 (장비 옵션 `[스킬명]` → invested 가산, 현재 `applyItemPassives` 동작 유지)
- **B. `item_passive_pool` 살리기**
  - 4개 패시브 동작을 `attack()` 안에 실제 구현
  - `getPassivePool` 을 진짜 `item_passive_pool` read 하도록 변경
  - `applyItemPassives` 의 스킬 invested 가산은 제거 또는 보조 역할

**리스크**: 기존 보유 장비의 옵션 텍스트가 의미를 잃을 수 있음. 마이그레이션 시 기존 옵션 처리 (유지/제거/재롤) 결정 필요.

**파일**: `lib/db/queries/items.ts`, `lib/db/seed.ts`, `lib/db/schema.ts` 마이그레이션, 잠재적으로 `lib/battle.ts:attack()`.

## 4. character.max_hp 의미 schema 주석

**현재**: `lib/character/stats.ts` helper jsdoc 에 effective max 계산 정책은 명시됨. 그러나 DB 컬럼 `character.max_hp` 자체가 base only 임은 schema.ts 주석에 없음.

**작업**: `lib/db/schema.ts` 의 `max_hp` 컬럼 위에 한 줄 주석.
```sql
-- base only: base_hp + VIT*vit_to_max_hp. effective max는 computeEffectiveStats(...).combatStats.max_hp 로 항상 재계산
max_hp INTEGER DEFAULT 110,
```
주의: schema 주석은 split(";")과 충돌 위험 — 세미콜론 미포함 확인.

## 7. applyItemPassives 가 max_skp cap 적용

**배경**: `lib/battle.ts:applyItemPassives` 가 장비 패시브 `[스킬명]` 을 invested 에 +1 가산할 때 `skill_table.max_skp` 상한을 무시. 장비를 많이 끼면 invested 가 max_skp 를 초과하면서 효과 계수가 의도 이상으로 증폭됨.

**작업**: 가산 후 `Math.min(invested + count, max_skp)` 로 캡.
```ts
return skills.map(s => ({
  ...s,
  invested: Math.min(s.max_skp ?? s.invested + count, s.invested + (counts[s.name] ?? 0)),
}))
```
`SkillData` 타입에 `max_skp` 없으면 추가.

**파일**: `lib/battle.ts`, 타입 정의 파일.

## 8. /api/skills PUT 트랜잭션 적용

**배경**: 스킬 invested 변경 PUT 이 트랜잭션 밖 read-modify-write. 동시 호출 시 race window — 한 쪽 변경이 lost update 가능.

**작업**: `lib/db/queries/_helpers.ts:tx` helper 사용해 read + UPDATE 를 한 트랜잭션으로 묶기.

**파일**: `app/api/skills/route.ts` (또는 `lib/db/queries/skills.ts` 의 saveSkillInvestments).
