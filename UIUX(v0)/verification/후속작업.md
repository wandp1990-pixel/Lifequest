## 1. `ADMIN_SECRET` env 등록 + UI 헤더 전송

**배경.** `/api/battle-config` PUT 과 `/api/skill-db` POST/PUT/DELETE 가 모두 fail-closed 정책으로 통일됨 (env 미설정 시 503 차단). 현재 운영 중인 production 에는 `ADMIN_SECRET`이 미설정이라 두 라우트의 쓰기가 모두 불가. 또한 UI 측 `SkillDbPanel` 은 헤더를 보내지 않아, env 를 등록하는 순간 패널의 쓰기 동작이 즉시 깨진다.

**작업.**
- `(1)` `ADMIN_SECRET` 값을 어떻게 정할지 결정. 단순 임의 문자열이면 OK. 향후 다중 관리자 확장 가능성 있으면 도입 시점부터 토큰 발급 패턴 고려
- `(2)` `npx vercel env add ADMIN_SECRET production` (preview/development 도 동일 값 또는 별도 값 등록)
- `(3)` UI 에서 헤더 주입 방식 결정:
  - 옵션 A: SettingsDrawer 에 admin 토큰 입력란 추가 → localStorage 저장 → SkillDbPanel/BattleConfigPanel fetch 호출 시 `Authorization: Bearer <token>` 헤더 추가
  - 옵션 B: 클라이언트가 직접 fetch 하지 않고, 한 단계 더 추상화된 server-action 또는 admin-only route 경유. 더 안전하지만 변경 범위 큼
  - 옵션 C: env 노출 (`NEXT_PUBLIC_ADMIN_SECRET`) — **권장 안 함**. 클라이언트에 secret 박힘
- `(4)` 패널 측 fetch helper 수정 (`SkillDbPanel.tsx` 의 모든 POST/PUT/DELETE 호출, `BattleConfigPanel` 동등 부분)

**예상 영향.** UI 두 패널만 영향. fetch 실패 시 토스트로 "관리자 토큰을 확인하세요" 안내 필요.

**파일.** `components/game/SettingsDrawer.tsx`, `components/game/SkillDbPanel.tsx`, `components/game/BattleConfigPanel.tsx` (있다면), `app/api/battle-config/route.ts` (변경 없음 — 이미 fail-closed).

## 2. `runBattle` 분해 + `trigger_condition` enum 화

**배경.** `lib/battle.ts:runBattle` 한 함수에 데미지 계산, 명중/회피, 치명타, 스킬 트리거(7~8개 분기), HP/MP 관리, 로그 누적이 모두 인라인. 함수 길이 200줄+. 스킬 trigger_condition 은 한국어 문자열 비교 (`"HP 25% 이하"`, `"매 3턴"` 등) 가 7곳에 흩어져 있어 한 글자 오타 시 silent fail.

**작업.**
- `(1)` `lib/skills/triggers.ts` 신규 — trigger_condition 문자열 상수화. seed.ts 와 battle.ts 가 동일 상수 참조하도록 import
  ```ts
  export const SKILL_TRIGGERS = {
    BATTLE_START:   "전투시작",
    FIRST_STRIKE:   "선공",
    EVERY_3_TURNS:  "매 3턴",
    ON_HIT:         "명중시",
    ON_CRIT:        "치명타시",
    ON_EVADE:       "회피시",
    ON_DEATH:       "사망시",
  } as const
  ```
- `(2)` `runBattle` 을 다음 함수로 분해:
  - `calcDamage(attacker, defender, isSkill)` — 데미지 + 치명타/명중/회피 판정
  - `applyTriggeredSkills(phase, ctx)` — 트리거 시점별 효과 적용 (dispatcher)
  - `runTurn(player, monster, ctx)` — 한 턴 처리
  - `runBattle` 은 상위 loop 만 남김
- `(3)` `seed.ts` 의 trigger_condition 값을 `SKILL_TRIGGERS` 상수로 교체 (런타임 시드 변경은 마이그레이션 필요 여부 확인)

**리스크.** 전투 로직 회귀가 사용자 체감 가장 큼. 변경 전 다음을 준비해야 함:
- runBattle 의 입력/출력 fixture 10~20개 (몬스터/플레이어/스킬 조합) 캡처
- 같은 입력으로 변경 전후 결과 동일성 비교 스크립트

**파일.** `lib/battle.ts`, `lib/skills/triggers.ts` (신규), `lib/db/seed.ts`. 가능하면 `lib/battle/` 디렉토리로 분리 후 `lib/battle.ts` 를 re-export shim 으로 유지.

## 3. `getPassivePool` SoT 정리 + `item_passive_pool` seed 재작성

**배경.** `lib/db/queries/items.ts:getPassivePool` 이 함수명과 다르게 `skill_table` 전체를 읽어 액티브 스킬도 가챠 패시브 풀에 포함시킴. `item_passive_pool` seed 의 4개 row(더블어택/생명흡수/방어무시/반사) 는 어떤 코드도 읽지 않는 dead 데이터. 더불어 그 4개 패시브는 `attack()` 내 변수가 0이라 사실상 비활성 상태 (`lib/battle.ts:308-309`).

**작업 — 두 갈래 정책 결정 필요.**
- **갈래 A**: 가챠 패시브 풀의 SoT 를 `skill_table` 로 확정
  - `getPassivePool` 을 `getGachaPassiveSource` 로 rename
  - `item_passive_pool` seed/마이그레이션 row 4개 제거 또는 `is_active=0`
  - 가챠 패시브 = 보유 스킬 강화 (장비 옵션의 `[스킬명]` 이 invested 에 가산)
- **갈래 B**: `item_passive_pool` 을 살리기
  - 4개 패시브 (더블어택/생명흡수/방어무시/반사) 의 실제 동작을 `attack()` 안에 구현 (현재 변수만 있고 사용처 없음)
  - `getPassivePool` 을 진짜 `item_passive_pool` 에서 read 하도록 변경
  - 갈래 A 의 장비 → 스킬 invested 가산 로직(`applyItemPassives`) 은 제거하거나 보조적 역할로 격하

**리스크.** 가챠 결과의 옵션 텍스트가 변경되어 기존 보유 장비의 옵션이 의미를 잃을 수 있음. 마이그레이션 시 기존 장비 옵션을 어떻게 다룰지(유지/제거/재롤) 결정 필요.

**파일.** `lib/db/queries/items.ts`, `lib/db/seed.ts` (item_passive_pool 부분), `lib/db/schema.ts` 마이그레이션, 잠재적으로 `lib/battle.ts:attack()`.

## 4. `character.max_hp` 의미 정책 명문화

**배경.** DB 컬럼이 base 인지 effective 인지 코드 위치마다 다름. 현재 동작:
- `gainExp` 가 저장하는 `max_hp` = base + VIT * vit_to_max_hp (아이템/스킬 제외)
- `/api/character` GET 의 `effective.max_hp` = base + 아이템 + 스킬 패시브 포함
- 다른 라우트(전투/장비)는 `char.max_hp` 직접 참조하지 않음

**작업.** schema.ts 의 `character.max_hp` 옆에 주석 추가:
```sql
max_hp INTEGER DEFAULT 110,  -- base only (= base_hp + VIT*vit_to_max_hp). effective max는 computeEffectiveStats(...).combatStats.max_hp 로 항상 재계산
```
+ `lib/character/stats.ts` 의 helper jsdoc 에 동일 정책 명시.

**파일.** `lib/db/schema.ts`, `lib/character/stats.ts`.

## 5. `/api/character` DELETE mode=full 의미 명확화

**배경.** `mode=full` 임에도 character 의 stat/level/exp/tickets 는 리셋 안 되고, 로그류와 equipment, skill_log 만 정리됨. `last_regen_at` 도 미리셋 → 직후 신규 항목 추가 시 회복 시계 어긋남.

**작업.** 두 옵션:
- 옵션 A: `mode=full` 에 character 의 능력치도 리셋 추가 (`UPDATE character SET level=1, total_exp=0, str=0, vit=0, ..., last_regen_at=now() WHERE id=1`)
- 옵션 B: 함수명/주석 변경. "데이터 로그만 초기화. 캐릭터 능력치 리셋은 별도 액션 (PUT)" 으로 의도 명문화

**파일.** `app/api/character/route.ts`.

## 6. 옵션 텍스트 ↔ `battle.ts` 파서 정합 강제

**배경.** `item_ability_pool.name` 한국어 문자열이 `lib/battle.ts:336-350` 의 if-else 체인과 정확 일치해야 보너스 적용. 새 능력 추가 시 파서 갱신 누락 = silent fail.

**작업.** 두 가지 방법:
- 방법 A: prebuild script — seed 데이터의 ability name 을 추출해 battle.ts 파서가 모두 처리하는지 검증, 누락 시 빌드 실패
- 방법 B: dev 모드 warning — 파서가 unknown key 만나면 `console.warn` 로깅. 운영 환경에선 silent

**파일.** `scripts/lint-ability-names.ts` (신규) 또는 `lib/battle.ts:buildPlayerCombatStats` 의 else 분기.

## 8. 회귀 방지 — 회귀 테스트 시나리오 (향후 작업 권장)
