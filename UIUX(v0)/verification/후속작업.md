## 2. runBattle 분해 + trigger_condition enum 화

**배경**: `lib/battle.ts:runBattle` 한 함수에 데미지 계산, 명중/회피, 치명타, 스킬 트리거 7~8개 분기, HP/MP 관리, 로그 누적이 모두 인라인 (200줄+). `trigger_condition` 은 한국어 문자열 비교 (`"HP 25% 이하"`, `"매 3턴"` 등) 가 7곳에 흩어져 있어 한 글자 오타 시 silent fail.

**작업**:
1. `lib/skills/triggers.ts` 신규 — trigger_condition 상수
   ```ts
   export const SKILL_TRIGGERS = {
     BATTLE_START:  "전투시작",
     FIRST_STRIKE:  "선공",
     EVERY_3_TURNS: "매 3턴",
     ON_HIT:        "명중시",
     ON_CRIT:       "치명타시",
     ON_EVADE:      "회피시",
     ON_DEATH:      "사망시",
   } as const
   ```
2. `runBattle` 분해:
   - `calcDamage(attacker, defender, isSkill)` — 데미지 + 치명타/명중/회피 판정
   - `applyTriggeredSkills(phase, ctx)` — 트리거 시점별 효과 dispatcher
   - `runTurn(player, monster, ctx)` — 한 턴 처리
   - `runBattle` 은 상위 loop 만 유지
3. `seed.ts` 의 trigger_condition 값을 `SKILL_TRIGGERS` 상수 참조로 교체 (런타임 시드 변경 시 마이그레이션 필요 여부 검토)

**사전 조건 (회귀 방지)**: 변경 전 fixture 10~20개 캡처 + 변경 전후 결과 동일성 비교 스크립트.
- 평타 명중 (STR 30, VIT 10 vs C급 슬라임)
- 평타 회피 (DEX 격차 큰 경우)
- 치명타 발생 (LUK 50 + 치명타 패시브)
- HP 25% 트리거 "역전"
- 매 3턴 트리거 (9턴 전투에서 3회 발동)
- 사망 시 SURVIVE
- 명중 시 EXTRA_HIT
- 패배 후 재도전 (`pending_battle_monster` 동일성)
- `restoreMode=half` 시작 HP 보장
- `restoreMode=none` `last_regen_at` 시계 유지

테스트 위치 후보: `lib/__tests__/battle.test.ts` (vitest 도입 동반).

**파일**: `lib/battle.ts`, `lib/skills/triggers.ts` (신규), `lib/db/seed.ts`. 가능하면 `lib/battle/` 디렉토리 분리 후 `lib/battle.ts` 를 re-export shim 유지.

## 후순위 (LOW, 필요 시점에)

- `gainExp` 트랜잭션 내 equipment read 동시성 — 단일 사용자 환경 안전. 멀티유저 확장 시 재검토
- `calcRegen` KST 하드코딩 — 다국가 확장 시점에
- `parseEquippedStatBonuses` 와 `buildPlayerCombatStats` 옵션 파싱 코드 중복 — 공통 파서 추출
- `SkillDetailSheet` 의 `SURVIVE` 등 `base_value` 의미없는 스킬도 "5.0%" 표시 — 효과 표시 룰 보강
- gacha 옵션 단위 (`%` vs `Pt`) 처리 명시 주석
- `item_slot_table.excluded` 의 dead 항목 "물리방어력" 제거 또는 ability_pool 매칭 추가
- `unit="HP"` / `unit="MP"` ability 의 `base_value` 가 `rollAbilityValue` 에서 무시되고 grade.stat_min/max 만 적용 — 의도 미구현인지 확인
- `monster_table` DROP — schema 에 dead 주석만 있는 상태, 검증 후 안전하게 제거 가능
- `item_passive_pool` 테이블 자체 DROP — 시드는 정리됐지만 빈 테이블 잔존. 마이그레이션 SQL (5번/9번) 의 UPDATE 도 의미 없음. prod DB 정리 시 함께 처리
