## 요약

- 발견된 이슈: **blocker 0 / high 2 / medium 5 / low 5**
- 핵심 권고
  1. `getPassivePool`이 `item_passive_pool`이 아니라 `skill_table`에서 read → 가챠 패시브 풀이 액티브 스킬까지 포함하고, "더블어택/생명흡수/방어무시/반사" 4개 핵심 패시브는 신규 가챠에서 영영 등장 못함. 의도 불일치이거나 의도였다면 변수명/주석이 거짓
  2. `/api/skill-db`가 인증 없이 스킬 정의 CRUD 가능 (GET/POST/PUT/DELETE 모두). 단일 사용자 서비스라 외부 위험은 낮으나, 실수로 노출된 URL 1번 호출만으로 데이터 파손
  3. `runBattle` 내부의 trigger_condition 문자열 비교가 7곳에 인라인 분산 + 한 글자 오타 시 silent fail. 상수/enum화로 컴파일 타임에 오타 검출 권장

## 발견사항

### [HIGH] 1. `getPassivePool`이 `skill_table`에서 read — 의도 불일치 가능성

- 위치: `lib/db/queries/items.ts:36-42`, `app/api/inventory/route.ts:91`, `lib/game/gacha.ts:32, 196-203`
- 현상:
  ```ts
  // lib/db/queries/items.ts:36-42
  export async function getPassivePool() {
    const db = getClient()
    const res = await db.execute(
      "SELECT id, name, description FROM skill_table WHERE is_active = 1"
    )
    return res.rows
  }
  ```
  함수 이름은 "PassivePool"인데 실제로는 `skill_table` 전체(액티브 8개 + 패시브 10개)에서 read한다. 가챠 결과 `[name]` 옵션에 "분노", "기사회생", "선제 강타" 같은 **액티브 스킬 이름**도 들어갈 수 있다.
- 시나리오:
  1. 가챠로 SR 등급 무기 획득 → passive 슬롯에 `[기사회생]`이 박힘
  2. `applyItemPassives`는 skill_table.name 기준 매칭으로 ACTIVE_SURVIVE_01의 invested에 +1 가산
  3. 사망 시 액티브 발동 가능 — 즉, "장비를 끼면 액티브 스킬이 1포인트 자동 투자된 효과"가 나옴
  4. 의도된 디자인이라면 이름이 거짓이고, 의도 아니라면 액티브 스킬을 가챠 풀에서 빼야 함
- 추가 증거 (`lib/db/seed.ts:221-234`): `item_passive_pool` 테이블에 더블어택/생명흡수/방어무시/반사 4개를 seed하지만, 가챠 코드는 이 테이블을 전혀 읽지 않으므로 dead seed. `lib/db/schema.ts:480-481, 532-535`의 마이그레이션도 dead row의 description을 갱신 중 — 작업이 모두 무효.
- 추가 증거 (`lib/db/schema.ts:622-627` `item_passive_to_skill_v1` 마이그레이션): 기존 장비의 `[더블어ték]` 등을 `[연속공격]`으로 일괄 치환했다는 이력이 있어, **의도는 분명히 "장비 옵션은 skill_table.name과 매칭되어야 한다"**. 즉 `getPassivePool`이 skill_table에서 read하는 건 "의도된 방향"이지만, 그 결과 액티브 스킬도 풀에 들어가고 item_passive_pool은 잔존하는 모순 상태
- 원인: 마이그레이션 단계에서 데이터 정합성만 맞추고, 풀 출처(item_passive_pool → skill_table 패시브 only) 일원화는 안 함
- 제안: 두 가지 중 택1
  - (a) `getPassivePool`을 `WHERE is_active=1 AND type='passive'`로 좁힘 → 액티브 스킬은 가챠로 안 나옴. item_passive_pool 테이블/seed는 schema.ts에서 제거. 명시적이고 깔끔
  - (b) 의도가 정말 "장비로 액티브를 1포인트 부여 가능"이었다면 함수명을 `getEquippableSkillsPool`로 rename + 주석 명시. 액티브 가챠가 게임 밸런스에 미치는 영향 별도 검토 필요

### [HIGH] 2. `/api/skill-db` 라우트 인증 없음

- 위치: `app/api/skill-db/route.ts` 전체 (32줄), 특히 `POST`/`PUT`/`DELETE`
- 현상: `withInit` 래퍼만 적용되어 있고 사용자/세션 검증 없음. 외부에서 `curl -X DELETE 'https://lifequest-bice.vercel.app/api/skill-db?id=ACTIVE_RAGE_01'` 한 번으로 스킬 정의 + 모든 사용자의 `skill_log`(invested_points)가 삭제됨 (`deleteSkillDb`가 skill_log → skill_table 순으로 cascade delete — `skills.ts:121-124`)
- 영향:
  - 단일 사용자 서비스 + URL이 공개되지 않은 한 실제 위험은 낮음
  - 그러나 `withInit`만 적용되었다는 사실이 보안 경계 정책 부재를 시사. `/api/character`도 동일하게 인증 없음 (모듈 4 검증에서는 GET/PUT 위주라 별도 플래그 안 함)
  - 향후 멀티 유저 또는 공개 시점에 모든 admin 라우트가 위험
- 원인: `withInit`은 initDb만 호출하고 인증은 별도 책임. 현재 admin 라우트와 일반 라우트의 구분이 코드상 없음
- 제안:
  - 단기: `/api/skill-db`, `/api/battle-config` 등 admin 라우트에 환경변수 기반 토큰 검증 추가 (예: `req.headers.get('x-admin-token') === process.env.ADMIN_TOKEN`)
  - 장기: `withAdmin` 래퍼 신설. admin 라우트 일괄 적용
- 우선순위: 현 시점에서는 medium에 가깝지만 데이터 손실 잠재성 때문에 high로 분류

### [MEDIUM] 3. `runBattle` trigger_condition 인라인 분산 — silent fail 위험

- 위치: `lib/battle.ts:615, 618, 649, 677, 691, 705, 734, 760` (8곳, 7개 트리거)
- 현상: 모든 분기가 한국어 문자열 동등 비교(`s.trigger_condition === "전투 시작"`). seed 한 글자 오타(예: "전투시작" → 공백 빠짐) 시 컴파일 통과, 런타임 silent fail. 어떤 정적 검사도 잡지 못함
- 영향:
  - 현재 seed와 비교 문자열은 100% 일치하므로 런타임 동작은 정상 (자동 비교에서 확인)
  - 그러나 seed.ts나 skill_table에 새 스킬 추가 시 trigger_condition 한 글자 오타 → 무발동, 디버깅 어려움
  - `/api/skill-db` PUT으로 trigger_condition 변경 시도 시도 동일 위험
- 원인: trigger_condition을 string 리터럴로 분산 처리. enum/상수화 안 됨
- 제안:
  ```ts
  // lib/skills/triggers.ts (신규)
  export const SKILL_TRIGGERS = {
    BATTLE_START:  "전투 시작",
    FIRST_STRIKE:  "선공 획득",
    HP_LOW_25:     "HP 25% 이하",
    EVERY_3_TURNS: "매 3턴",
    ON_CRIT:       "치명타 시",
    ON_HIT:        "명중 시",
    ON_EVADE:      "회피 시",
    ON_DEATH:      "사망 시",
  } as const
  export type SkillTrigger = typeof SKILL_TRIGGERS[keyof typeof SKILL_TRIGGERS]
  ```
  + `runBattle`에서 `s.trigger_condition === SKILL_TRIGGERS.BATTLE_START` 형태 사용. 추가로 seed 단계에서도 import해서 일관성 확보. effect_code도 동일 패턴 권장

### [MEDIUM] 5. `applyItemPassives`가 invested에 +1만 가산 — 불공정 강함

- 위치: `lib/battle.ts:48-59`
- 현상:
  ```ts
  for (const raw of equippedOptions) {
    ...
    counts[line.slice(1, -1)] = (counts[line.slice(1, -1)] ?? 0) + 1
  }
  return skills.map(s => ({ ...s, invested: s.invested + (counts[s.name] ?? 0) }))
  ```
  같은 옵션 `[기사회생]`이 두 장비에 동시 박혀 있으면 invested +2 가산. 가챠 자체는 한 장비당 패시브 1개로 제한되지만(`gacha.ts:198-203`), 여러 장비에 동일 옵션이 박힐 가능성 존재
- 영향:
  - skill의 max_skp가 20인데 장비로 +2~+9까지 추가 가능 (현재 9개 슬롯). 사용자는 SKP 0포인트 투자해도 장비만으로 max_skp 절반 효과 획득 가능
  - 장비 패시브가 max_skp cap을 무시함 — 의도된 디자인인지 명시 없음
- 원인: cap 검사 누락. invested 합산 후 `Math.min(s.max_skp, ...)` 같은 보호 없음
- 제안: 정책 결정 후 분기
  - 의도가 "장비 패시브는 무한 누적" → 현 코드 + 주석 추가
  - 의도가 "max_skp 초과 불가" → `Math.min(s.max_skp, s.invested + counts[s.name])` 적용
  - 의도가 "장비 1슬롯만 카운트" → counts를 Set으로 변경

### [MEDIUM] 6. `/api/skills` PUT의 부분 검증 / race window

- 위치: `app/api/skills/route.ts:22-50`
- 현상: 흐름이 (1) char + currentSkills read (2) skpDelta 계산/검증 (3) saveSkillInvestments (4) updateCharacter. (1)~(3)이 트랜잭션 밖이라 동시에 다른 PUT이 들어오면 SKP 이중 차감 가능
- 영향: 단일 사용자 환경 + UI는 단일 save 버튼이라 동시 호출은 거의 없음. 그러나 빠른 더블 탭이나 네트워크 재시도 시 race window 존재
- 원인: skill PUT이 트랜잭션 밖에서 read-modify-write
- 제안: `tx()` 헬퍼로 묶거나, character.skill_points UPDATE를 conditional UPDATE (`WHERE skill_points >= ?`)로 변경해 race 방어. inventory POST의 `WHERE draw_tickets >= ?` 패턴 차용 가능

### [LOW] 8. `ACTIVE_REVERSAL_01` HP_HEAL 분기의 MP 부족 처리 — `break` vs `continue`

- 위치: `lib/battle.ts:647-664`
- 현상:
  ```ts
  if (hpRatio <= 0.25) {
    for (const s of activeSkills) {
      if (s.trigger_condition !== "HP 25% 이하" || s.effect_code !== "HP_HEAL") continue
      if (skillUsed.has(s.id)) continue
      if (s.invested <= 0) continue
       const actualMpCost = getSkillMpCost(s)
       if (playerMp - mpCost < actualMpCost) break   // ← MP 부족이면 루프 중단
       ...
       break
    }
  }
  ```
  HP_HEAL 스킬이 여러 개 등록되어 있고 첫 번째가 MP 부족이면 두 번째도 발동 못 함. 현재 seed에서는 ACTIVE_REVERSAL_01 1개뿐이라 영향 없으나, 두 개 추가 시 의도와 다르게 동작 가능
- 원인: 여러 HP_HEAL 스킬을 순차 시도하려는 의도라면 `continue`여야 함. 단일 스킬만 trigger 의도라면 OK
- 제안: 현 단계 수정 불필요. 향후 HP_HEAL 스킬 추가 시 정책 결정 + 주석 추가

### [LOW] 9. `ACTIVE_REVERSAL_01` 처리 위치 — 공격 전 처리라 회복 후 행동 못 함

- 위치: `lib/battle.ts:642-667` (HP_HEAL 처리는 turn 시작 직후, 자신의 공격 전)
- 현상: 회복 후 같은 턴에 일반 공격은 진행됨. SKILL_INFO 설명("발동 시 즉시 한 번 더 공격" 같은 추가 행동 언급은 없음) 기준 정합. 단 `mp_cost` 계산 흐름이 후속 액티브와 한 mpCost 변수를 공유 — 같은 턴에 HP_HEAL + 매 3턴 + 명중 시 등이 모두 발동하면 mpCost 누적 후 한 번에 차감
- 영향: 한 번에 4-5개 액티브가 동시 트리거되는 케이스(턴 3, 6, 9 + HP 25% 이하 + 치명타 + 명중) 시 MP 동시 차감으로 일부 못 발동. 현 밸런스에서 충분히 발생 가능
- 원인: turn 단위로 mpCost 누적, 차감은 turn 끝에 1회 (`battle.ts:724`)
- 제안: 의도된 디자인이지만 명시 안 됨. 주석 추가 권장 ("같은 턴 내 액티브 스킬은 MP 풀을 공유하며, 부족하면 후순위 스킬 미발동")

  ```ts
  const unit = skill.effect_code.endsWith("_FLAT") ? "" : "%"
  ```
- 현상: `HP_HEAL`, `EXTRA_HIT`, `SURVIVE`도 `_FLAT`이 아니므로 `%` 표시. `SURVIVE` 스킬은 base_effect_value=5/coeff=2.25 → "5.0%" 표시되는데 이 5는 % 아님 (생존 발동에 의미 없는 base값일 가능성). 실제로 SKILL_INFO에서 `effect_explanation`에 "MP 비용이 없으며, 한 전투에서 한 번만 발동된다"라고만 적혀 있어 5라는 수치는 사용자에게 혼란
- 영향: UI 잘못된 정보 — "기사회생" 효과치를 5.0% → 6.25% → 7.5% ... 처럼 표시하지만 실제 동작은 발동 여부만 (HP를 1로 set, value 무관: `battle.ts:736`)
- 원인: effect_code별 단위/표시 형식이 한 곳에 정의되지 않음
- 제안: `effect_code → { unit, suffix, displayMode }` 매핑 테이블 신설하거나, base_effect_value의 의미가 없는 effect_code는 SkillDetailSheet에서 효과치 카드 자체 비표시 (조건부 렌더)

### [LOW] 12. `skill_log` 외래키 cascade 부재

- 위치: `lib/db/schema.ts:197-203`
  ```sql
  CREATE TABLE IF NOT EXISTS skill_log (
      id              INTEGER PRIMARY KEY AUTOINCREMENT,
      skill_id        TEXT REFERENCES skill_table(id),
      ...
  );
  ```
- 현상: FK 선언 있으나 `ON DELETE CASCADE` 없음. SQLite는 기본적으로 FK 검증도 PRAGMA로 켜야 활성화. Turso/libsql 환경에서 FK가 실제로 enforce되는지는 별도 확인 필요
- 영향: `deleteSkillDb`가 명시적으로 두 단계 DELETE 수행 (`skills.ts:121-124`)하므로 현 코드에서는 누수 없음. 그러나 향후 직접 SQL 삭제 시 orphan skill_log 발생 가능
- 제안: `ON DELETE CASCADE` 추가 또는 `deleteSkillDb` 외 경로 차단 명문화
